## 1. 简介

### 1.1 目的

本文定义 Graph-autofusion 仓库中使用 Agent 开发的通用协作模式，适用于 SuperKernel、Autofuse、MLIR 迁移、测试开发、构建脚本和文档维护等工作。目标是让 Agent 可以持续参与设计、编码、验证和文档沉淀，同时不影响主线稳定性、不污染用户工作区、不把未经验证的结论当作事实。

核心原则：

- Agent 可以辅助开发，但必须在明确任务边界内工作。
- 本地能写代码，不代表编译通过。
- Docker 能编译，不代表 NPU 真机通过。
- 任何结论都必须绑定环境、命令和证据。

### 1.2 范围

本文覆盖：

- 分支和工作区约束。
- Agent 任务输入/输出约束。
- 本地开发环境、Docker 编译验证环境、NPU 实测环境的边界。
- 依赖、镜像、CANN Toolkit 和 PyAsc 使用约束。
- 验证门禁、提交前检查和回退要求。

本文不覆盖：

- 具体特性的实现细节。
- 具体 tracker 任务拆分结果。
- CI 平台的具体流水线语法。
- NPU 服务器账号、地址、token、私有路径等敏感信息。

## 2. 总体概述

### 2.1 软件概述

Graph-autofusion 后续开发需要在长期迭代中同时保证两件事：

- 主线稳定性不受影响。
- 新能力可以在隔离分支、隔离目录和显式开关下持续推进。

因此 Agent 开发模式采用：

```text
本地开发环境
  -> Docker 编译验证环境
  -> NPU 实测环境
  -> 提交前复核
```

每一层只证明自己能证明的结论。不能跨层替代。

### 2.2 产品环境介绍

- 本地开发环境可以是 Mac、Windows 或 Linux，用于读代码、编辑、写文档、写轻量脚本。
- Docker 编译验证环境必须是 Linux，用于标准构建、UT/ST、专项构建和 CI 对齐验证。
- NPU 实测环境必须具备 CANN Toolkit、NPU driver、设备节点和真实用例输入，用于 `ascir_tool`、精度校验和端到端验证。
- 任务所需的重型依赖可以由团队预编译镜像提供，也可以由开发者通过本地路径显式指定。
- Docker 编译验证环境支持两种来源：用户自定义 Docker 环境，以及项目提供的一键构建 Docker 环境。
- CANN Toolkit 不内置进通用编译镜像，必须由外部环境注入。

### 2.3 软件功能

本文要求 Agent 在每个开发任务中产出以下信息：

- 任务目标和不做范围。
- 涉及文件和目录。
- 使用环境：本地、Docker、NPU 中的哪一层。
- 验证命令和预期结果。
- 实际验证证据。
- 风险、未验证项和回退方式。

### 2.4 设计约束

- 新需求默认从主线创建隔离开发分支；分支名按任务或项目约定执行。
- 长周期开发分支每天开始开发前，或开始新任务前，必须先执行受控主线同步检查点。
- 新代码优先集中在任务所属模块的隔离目录；只在必要入口做薄接线。
- 构建、测试、打包脚本的既有语义不得被复用或暗改。以 MLIR 迁移为例，`--no-autofuse` 仍表示跳过整个 Autofuse 后端构建，不能复用为 MLIR 开关。
- 新能力默认关闭，必须通过显式开关进入；默认构建路径不得引入新重型依赖。
- 所有构建命令限制并行度，优先 `-j 8`。
- Agent 不自动提交、不自动 push、不自动改写历史，除非用户明确要求。

### 2.5 假设和依赖关系

- 开发者可以在 Mac/Windows/Linux 本地编辑代码。
- 标准编译验证以 Linux Docker 或 CI Linux runner 为准。
- NPU 实测需要独立 NPU host 或 CI NPU runner。
- 预编译镜像、CANN Toolkit 注入方式和 NPU 访问方式由团队另行维护。
- 项目后续提供一键构建 Docker 环境的脚本或文档；在脚本落地前，允许开发者使用自定义 Docker 环境完成等价验证。

## 3. 需求分析与设计

### 3.1 整体介绍

Agent 开发按任务卡推进。任务卡是 Agent 的最小执行单元，必须包含：

```text
任务名称：
目标：
不做范围：
涉及文件：
允许修改目录：
需要环境：
验证命令：
验收标准：
回退方式：
```

tracker 任务编号只是管理粒度，不是固定工程边界。如果实际推进中发现任务不合理，可以修改、合并、拆分、增加或删除，但必须说明原因、影响范围和新的验收口径。

### 3.2 功能需求 1：分支和工作区保护

1. 介绍

   Agent 开始任何代码或文档修改前，必须确认分支、工作区状态和用户已有改动。

2. 输入

   - 当前分支。
   - `git status --short`。
   - 用户指定任务。

3. 处理

   - 如果不在预期分支，先报告，不直接切换。
   - 如果存在无关脏文件，记录并避开。
   - 如果目标文件已有用户改动，先读取 diff，基于现状继续编辑，不覆盖。

4. 输出

   - 本次计划修改的文件清单。
   - 不会触碰的已有脏文件说明。

#### 3.2.1 每日同步主线检查点

长期开发分支需要高频跟官方主线对齐，但不能把同步做成无条件自动任务。Agent 每天开始开发前，或开始一个新任务前，必须执行一次受控同步检查点。

同步前检查：

```bash
git status --short --branch
git remote -v
git fetch upstream
git branch -r --list 'upstream/main' 'upstream/master'
```

约束：

- 工作区不干净时，只报告差异和风险，不执行 `rebase` 或 `merge`。
- 官方主线分支以仓库实际配置为准；当前仓库优先使用 `upstream/master`。
- 个人开发分支可以使用 `git rebase upstream/master`，保持历史线性。
- 多人共享分支优先使用 `git merge upstream/master`，避免改写他人基于该分支的历史。
- 发生冲突时停止自动处理，报告冲突文件、冲突类型和建议处理方式。
- 不使用 `git reset --hard`、`git checkout --` 等破坏性命令清理冲突或工作区。
- 不自动 force push。只有用户明确要求同步远端重写历史时，才允许使用 `--force-with-lease`。

同步后检查：

```bash
git status --short --branch
git log --oneline --decorate -5
```

如果同步引入了代码或构建相关变化，按任务风险运行必要 baseline。文档任务至少执行：

```bash
git diff --check
```

构建或测试相关任务按影响范围执行，例如：

```bash
sh build.sh --pkg -j 8
sh build.sh --pkg --no-autofuse -j 8
```

同步报告格式：

```text
主线同步检查：
当前分支：
官方主线：
工作区状态：
同步方式：
结果：
冲突/未处理：
验证：
```

### 3.3 功能需求 2：环境分层

#### 3.3.1 本地开发环境

本地开发环境用于：

- 读代码。
- 改代码。
- 写文档。
- 写测试。
- 运行轻量脚本、格式检查、静态文本检查。

适配平台：

- Mac。
- Windows。
- Linux。

约束：

- 不要求安装完整编译依赖、CANN Toolkit 或 NPU 设备环境。
- 不要求能完整编译 Autofuse。
- 不能用本地开发环境声明“编译通过”。
- 不能用本地开发环境声明“NPU 真机通过”。
- 本地生成的临时产物不得进入版本库。

允许结论：

```text
已完成代码编辑。
已完成文档更新。
已完成静态检查。
```

禁止结论：

```text
编译通过。
UT/ST 通过。
NPU 实测通过。
```

#### 3.3.2 Docker 编译验证环境

Docker 编译验证环境用于：

- Linux 标准构建。
- 默认包验证。
- `--no-autofuse` 兼容验证。
- 专项显式开关构建验证。
- UT/ST/lit/FileCheck。
- CI 流水线对齐。

适配模式：

- Mac 本地开发 + Linux Docker 编译验证。
- Windows 本地开发 + Linux Docker 编译验证。
- Linux 本地开发 + Linux Docker 编译验证。
- CI 使用同源镜像验证。

环境来源：

- **用户自定义 Docker 环境**：开发者可以使用已有 Linux Docker 镜像，只要镜像满足编译依赖、ABI、Python、CMake/Ninja、专项依赖等要求，并在验证报告中记录镜像 tag/digest、关键工具版本和依赖路径。
- **项目一键构建 Docker 环境**：项目提供脚本或文档，从 Dockerfile 构建推荐编译验证镜像，统一基础编译器、Python 依赖、常用工具和专项依赖版本，降低 Mac/Windows 开发者配置成本。

两种来源的环境都必须满足同一套验证门禁。不能因为使用自定义环境而降低验证要求，也不能因为使用项目一键镜像而省略版本记录。

镜像内容建议：

- 项目基础编译依赖和任务专项依赖。
- CMake/Ninja。
- Python 依赖。
- lit/FileCheck。
- ccache。
- 基础编译器。

镜像不内置：

- CANN Toolkit。
- NPU driver。
- 项目源码。
- 私有账号、token、密钥。
- 用户私有路径。

基础验证命令：

```bash
sh build.sh --pkg -j 8
sh build.sh --pkg --no-autofuse -j 8
```

配置专项依赖后验证，以 MLIR 任务为例：

```bash
sh build.sh --pkg --enable-mlir -j 8
```

允许结论：

```text
Linux Docker 默认构建通过。
Linux Docker --no-autofuse 构建通过。
Linux Docker 专项开关构建通过。
```

禁止结论：

```text
NPU 真机通过。
性能达标。
现网可用。
```

#### 3.3.3 NPU 实测环境

NPU 实测环境用于：

- `ascir_tool` 真实 NPU E2E。
- kernel 编译、launch、输出比对。
- 精度校验。
- 性能回归。
- 迁移 slice 的最终行为确认。

环境要求：

- CANN Toolkit 可用，并显式设置 `ASCEND_HOME_PATH`。
- NPU driver 和设备节点可用。
- 显式指定 `ASCEND_DEVICE_ID`。
- 输入、golden、dump、日志路径可归档。

实测记录必须包含：

```text
源码 commit：
镜像 tag/digest：
ASCEND_HOME_PATH：
ASCEND_DEVICE_ID：
测试 case：
执行命令：
输出路径：
结果摘要：
```

允许结论：

```text
指定 case 在指定 NPU 环境下通过。
指定 dump 与 golden 一致。
指定性能数据已采集。
```

禁止结论：

```text
全场景通过。
所有 NPU 环境通过。
默认路径可切换。
```

除非对应范围全部验证完成。

### 3.4 功能需求 3：依赖和镜像约束

1. 任务专项依赖

   - 可以使用团队提供的预编译镜像。
   - 可以通过任务约定的环境变量或 CMake 参数指定本地已编译产物。
   - Agent 不允许默认联网 clone 或静默编译重型依赖。
   - 示例：MLIR 迁移任务可以通过 `LLVM_BUILD_DIR` 或 `AF_LLVM_ROOT` 指定 LLVM/MLIR 产物。

2. Docker 环境

   - 支持用户自定义 Docker 环境。
   - 支持项目一键构建 Docker 环境。
   - 自定义环境必须在验证报告中记录镜像来源、tag/digest、OS、编译器、CMake/Ninja、Python、专项依赖路径和 ABI 信息。
   - 一键构建环境必须由项目内 Dockerfile/脚本/README 描述构建方式，构建结果也必须记录 tag/digest 和 manifest。
   - 两种环境都不得内置密钥、token、用户私有路径、NPU driver 或项目源码。

3. CANN Toolkit

   - 不放入通用编译镜像或专项依赖镜像。
   - 通过 host、CI secret、volume 或环境变量外部注入。
   - Agent 不记录私有安装路径到正式文档，除非用户明确要求。

4. 任务私有第三方依赖

   - 正式依赖不得指向个人仓。
   - 如需维护第三方源码差异，优先采用官方仓 + 本仓 patchset。
   - patch 开发和导出必须可复现、可 review。
   - 示例：MLIR 迁移中的 PyAsc 差异应使用官方 PyAsc 仓 + Autofuse 仓内 patchset。

### 3.5 功能需求 4：验证和结论约束

Agent 的结论必须使用以下格式：

```text
环境：
命令：
结果：
未覆盖：
```

示例：

```text
环境：Linux Docker 编译验证环境
命令：sh build.sh --pkg -j 8
结果：通过
未覆盖：NPU 真机执行未验证
```

禁止写法：

```text
应该没问题。
理论上能跑。
本地看起来可以。
```

### 3.6 功能需求 5：提交和回退约束

- Agent 不主动提交。
- 用户明确要求提交前，Agent 必须展示：
  - `git diff --stat`
  - 涉及文件列表
  - 关键 diff 摘要
  - 验证命令和结果
  - 未验证项
- 提交粒度必须和任务粒度一致。
- 不把无关文档、临时产物、截图、日志、build 目录混入提交。
- 回退时只回退 Agent 自己的改动，不回退用户已有改动。

## 4. 非功能需求

### 4.1 可维护性

- 规范文档作为 Agent 开发前置读物。
- 每个任务卡都有明确范围和退出条件。
- 环境差异通过“本地开发 / Docker 编译验证 / NPU 实测”三层隔离，减少口径混淆。

### 4.2 可测试性

- 本文不新增可执行代码。
- 后续可为 Docker/NPU 验证封装统一脚本，但脚本必须输出环境、命令、结果和未覆盖项。

### 4.3 可移植性

- 支持 Mac/Windows 本地开发。
- 标准编译统一落到 Linux Docker。
- NPU 实测统一落到具备 CANN 和设备节点的 Linux host。

### 4.4 可靠性

- 通过分支、目录、开关、环境和验证门禁降低 Agent 误改风险。
- 通过不自动提交、不静默联网、不覆盖用户改动降低工作区风险。

### 4.5 特性交叉影响

| 场景 | 适用性 | 分析说明 |
|------|--------|----------|
| SuperKernel Python 接口 | 不适用 | 本文不修改 SuperKernel Python 包、选项解析或 pytest 用例。 |
| SuperKernel C++/AOT 接口 | 不适用 | 本文不修改 SuperKernel C++ 接口、AOT 测试或 ABI/API。 |
| Autofuse 图优化 | 部分适用 | 本文要求后续涉及图优化时必须明确图改写等价性和验证命令，但本文不改 optimize 逻辑。 |
| Autofuse Codegen/Backend | 适用 | Codegen/backend 相关结论必须经过 Docker 编译和 NPU 实测分层验证；MLIR 迁移只是其中一个适用场景。 |
| AscendC API / Runtime 交互 | 部分适用 | 本文不新增 runtime 调用，但定义 NPU 实测环境中必须记录 CANN、device、case 和结果。 |
| Python/C++ 混合绑定 | 部分适用 | Agent 修改 Python/C++ 绑定时必须进入 Docker 编译验证；本文不修改绑定代码。 |
| 构建与打包 | 适用 | 本文要求默认构建、兼容路径和专项显式构建分别验证，且构建命令限制 `-j 8`。 |
| 测试与覆盖率 | 适用 | 本文定义 UT/ST/lit、`ascir_tool`、NPU E2E 的环境边界。 |
| 性能与日志 | 部分适用 | 本文不新增日志；性能结论只能来自 NPU 实测或专门 benchmark。 |
| 兼容性 | 适用 | 本文要求不改变 `--no-autofuse`、默认 build、run 包边界和现有用户路径。 |

## 5. 性能

### 5.1 编译时长

Agent 触发构建时必须限制并行度，优先 `-j 8`。LLVM/MLIR 等重型依赖不在普通任务中从源码静默编译，避免 OOM 和长时间占用。

### 5.2 执行性能

本文不引入运行时代码。任何性能结论必须来自 NPU 实测环境或专门 benchmark，不能由本地开发或 Docker 编译验证推断。

### 5.3 内存和产物大小

Docker 镜像、重型依赖 tarball、build 目录、coverage、kernel_meta、dump 日志均属于环境或临时产物，不默认进入版本库。

## 6. 接口设计

### 6.1 新增/修改接口描述

本文不新增 C++/Python 对外接口。新增的是 Agent 协作过程接口，即任务卡和验证结论格式。

任务卡接口：

```text
任务名称：
目标：
不做范围：
涉及文件：
允许修改目录：
需要环境：
验证命令：
验收标准：
回退方式：
```

验证结论接口：

```text
环境：
命令：
结果：
未覆盖：
```

### 6.2 接口检查项

| 检查项 | 子检查项 | 是否涉及 | 说明 |
|--------|----------|----------|------|
| 接口说明 | 是否需要接口评审 | 否 | 不新增代码接口。 |
| 接口说明 | 是否需要补充文档 | 是 | 本文即协作过程文档。 |

## 7. 软件设计

### 7.1 关键数据结构

无代码数据结构。过程数据结构为：

- 任务卡。
- 环境记录。
- 验证记录。
- 提交前检查记录。

### 7.2 关键技术/算法

无算法。关键机制是环境隔离和证据门禁。

### 7.3 流程设计

```text
用户提出任务
  -> Agent 读取规范和相关设计
  -> 执行每日同步主线检查点
  -> 输出任务卡
  -> 本地开发环境修改
  -> Docker 编译验证环境验证
  -> 需要时进入 NPU 实测环境
  -> 汇报验证证据和未覆盖项
  -> 用户确认后再提交或继续调整
```

### 7.4 对子模块的修改

本文仅新增规范文档，不修改源码模块。

### 7.5 错误处理

#### 系统错误

- Docker 不可用：停止编译结论，只汇报本地开发完成和 Docker 未验证。
- 专项依赖缺失：停止对应专项构建结论，提示按任务文档配置依赖路径。示例：MLIR 迁移任务提示配置 `LLVM_BUILD_DIR` 或 `AF_LLVM_ROOT`。
- CANN 缺失：停止 NPU 实测结论，提示缺少 `ASCEND_HOME_PATH` 或 CANN 环境。
- NPU device 不可用：停止真机结论，记录 device 和错误信息。

#### 接口错误

- 用户任务边界不清：Agent 先列出可能理解并请求确认。
- 工作区有冲突改动：Agent 先报告，不覆盖。
- 主线同步冲突：Agent 停止自动处理，报告冲突文件、冲突类型和建议处理方式。
- 验证失败：Agent 报告失败命令、关键错误和下一步定位建议，不宣称完成。

## 8. 安全检查

- 不硬编码密码、token、账号、私有服务器地址或私有安装路径。
- 不把 CANN/NPU 服务器信息写入正式文档。
- 不静默联网下载依赖。
- 不使用 `git reset --hard`、`git checkout --` 等破坏性命令回退用户改动。
- 不在工作区不干净时自动执行 `rebase`、`merge` 或历史改写。
- 不提交 build、coverage、kernel_meta、dump、日志、截图等临时产物。
- 不改变 `--no-autofuse` 语义。
- 不在默认构建路径引入任务专项重型依赖。示例：MLIR 迁移默认路径不得引入 LLVM/MLIR/PyAsc。

---

## 9. 兼容性检查

- 兼容 Mac/Windows/Linux 本地开发。
- 编译验证统一通过 Linux Docker，避免宿主机差异污染结果。
- NPU 实测必须在真实 NPU 环境完成，避免把 Docker 编译结果误认为真机结论。
- 不影响现有 `build.sh` 参数、run 包布局、Python 包、C++ ABI/API。

## 10. 测试设计

### 10.1 测试边界

本文作为流程规范，不需要 UT。后续若把流程脚本化，需要覆盖：

- 默认构建验证。
- `--no-autofuse` 验证。

### 10.2 测试用例设计

| 测试类别 | 关键测试项 | 测试方法 | 用例类型 |
|----------|------------|----------|----------|

| 功能 | NPU 实测记录 | `ascir_tool` case | E2E |

sh build.sh --pkg --no-autofuse -j 8

专项依赖配置后，以 MLIR 任务为例：

```

NPU 环境中按 case 执行：

bash autofuse/tests/st/codegen/ascir_tool/test_ascir.sh --mode=0 --case=<case_name>
```

## 11. 验收标准

- Agent 开发前能明确任务卡。
- Agent 每天开始开发前或开始新任务前能执行主线同步检查点，并在工作区不干净或冲突时停止自动处理。
- Agent 汇报结论时能区分本地开发、Docker 编译验证和 NPU 实测。
- 默认路径、兼容路径、专项显式开关的验证边界清晰。
- Mac/Windows 本地开发 + Linux Docker 编译验证的模式被明确支持。
- 用户自定义 Docker 环境和项目一键构建 Docker 环境都被明确支持，且验证门禁一致。
- CANN Toolkit 和 NPU driver 不进入通用编译镜像或专项依赖镜像。
- Agent 不再用本地编辑结果替代编译结论，也不再用 Docker 编译结果替代 NPU 真机结论。
