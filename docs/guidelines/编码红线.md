# 编码红线

以下规则用于 Graph-autofusion 需求设计、编码和代码检视。触碰红线的修改必须退回或重新设计。

## 通用规则

### 1. 禁止硬编码敏感信息和高风险敏感词

不得在源码、脚本、配置或测试数据中硬编码密码、密钥、Token、账号、公网 IP、域名、邮箱或用途不明的大数组。测试需要的假数据必须明显为无效占位值。

### 2. 外部输入作为索引、长度或偏移前必须校验

任何来自用户、模型、图属性、配置、Python 入参、文件或 runtime 返回的数据，在作为数组索引、内存长度、指针偏移、循环次数或 shape 维度前必须检查范围。

### 3. 整数运算必须防止溢出、反转和除 0

涉及 shape、tensor size、buffer size、tiling size、block 数、线程数、字节数和偏移的整数运算必须检查边界。运算结果用于内存申请或拷贝时必须尤其谨慎。

### 4. 资源释放必须覆盖异常分支

内存、文件句柄、动态库句柄、runtime 资源、stream/event、临时目录、Python 引用计数等资源必须明确所有权和释放路径。异常分支不得泄露资源。

### 5. 内存申请前必须判断大小，申请后必须校验成功

不得申请 0 长度或未校验的大块内存。外部 allocator 或 runtime 分配返回后，必须检查指针和状态码。

### 6. 禁止依赖未指定求值顺序或未定义行为

不得编写 `func(a++, a)` 等依赖参数求值顺序的代码，不得使用悬空引用、越界访问、未初始化变量或类型别名违规访问。

### 7. 禁止无理由扩大修改范围

不得借需求修改无关模块、格式化无关文件、顺手重构或删除既有代码。发现无关问题应记录说明，除非用户明确要求处理。

## Graph-autofusion 规则

### 1. 修改对外接口时必须评估 API/ABI 兼容

动态库导出的 C/C++ 接口、Python 绑定、脚本参数、配置项、run 包安装路径和 wheel 内容都属于兼容边界。修改接口签名、结构体布局、枚举值、默认参数或产物命名前必须说明兼容影响。

### 2. 禁止在代码中硬编码芯片类型、平台类型或框架类型

不得通过硬编码芯片型号、SoC 字符串、前端框架名称或网络名称决定核心逻辑。确需区分能力时，应通过 CANN/runtime/平台能力接口查询，并说明兼容策略。

### 3. 图改写必须保持数据边和控制边等价

Autofuse pass 插入、删除、替换或重排节点时，必须同时考虑数据依赖和控制依赖。不得只验证输出 tensor 而忽略执行顺序约束。

### 4. 新增有时序依赖的 pass 必须说明依赖原因

新增或调整 pass 顺序时，必须在设计和代码注释中说明依赖的前置条件、后置影响和无法解耦的原因。

### 5. 图改写结果必须确定

对图做增删改操作时，禁止依赖 `std::unordered_map` 遍历顺序或对象指针地址顺序生成最终图结构。需要稳定顺序时使用节点名、拓扑序或显式排序。

### 6. 高调用频率路径禁止新增默认开启的海量日志

编译期核心循环、运行时调度、kernel launch、内存拷贝和每算子路径中不得新增默认开启的大量日志。必须新增时，说明开关、等级和性能影响。

### 7. 调用 `rt*` / `aclrt*` / AscendC 接口必须满足生命周期约束

同步拷贝调用时源和目的内存必须立即有效；异步拷贝调用后，host/device 内存必须保持到任务实际执行完成。调用前应检查 size > 0，调用后检查返回值。

### 8. 禁止使用非开放 runtime 接口

`rt` 接口只能使用 CANN 对外头文件中声明的开放接口。非开放接口不得绕过封装直接调用。

### 9. Python/C++ 绑定必须正确处理引用计数和异常

CPython 扩展或 pybind 相关代码必须明确对象所有权、引用计数、GIL、异常转换和错误返回。不得吞掉底层错误。

### 10. 构建脚本不得破坏增量构建和离线构建

CMake、`build.sh`、第三方依赖下载和打包逻辑修改后，必须考虑首次构建、增量构建、离线依赖、`--no-autofuse` 和指定输出路径场景。

### 11. 测试临时产物不得进入版本库

测试生成的代码、日志、coverage、build、run 包、临时 kernel_meta 等产物必须位于已忽略目录，不得提交。

### 12. 全局/静态对象析构不得依赖跨 SO 调用

全局对象、静态单例或延迟析构对象不得在进程退出阶段触发跨动态库资源释放。需要清理时提供显式 Finalize/Cleanup 流程。
